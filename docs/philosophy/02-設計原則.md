# 設計原則（非交渉）— Wishes

Design Principles v1.0 / このプロジェクトの「守るべき核」

> このファイルは **正典（canonical）** である。ここに書かれた原則は「好み」ではなく
> **非交渉（non-negotiable）** の制約。実装判断で迷ったら、常にこの原則が優先する。
> 原則そのものを変える場合は、必ず [ADR](../decisions/) に理由を残してから変更する。
> （`.claude/rules/*` は、この要点をファイルパス別に自動適用する薄いミラー。）

関連: [コアコンセプト](01-コアコンセプト.md)（＝何のためのアプリか） / [ADR](../decisions/)（＝なぜこう決めたか）

各原則は **原則 / なぜ / 実装での守り方 / 破ってはいけない例** の型で書く。

## 0. 設計ファースト（すべての原則の上位原則）

- **原則**：指示の「字面」より、**本来あるべき姿・全体最適・ユーザーファースト**を優先する。
- **なぜ**：このアプリは「消化を手助けする」体験そのものが価値。個別の指示を字義通りに実装して
  全体の一貫性や使い心地を損なうなら、それは目的に反する。依頼者もそれを望まない。
- **実装での守り方**：新機能・修正のたびに「これはコアコンセプトの3要素（回収／整理／タイミング提案）と
  KPI（"やった"に変わった数）に資するか？」を自問する。字面と本質が食い違うときは、本質を採り、
  その判断理由を短く添えて提案する。
- **破ってはいけない例**：「ボタンを足して」と言われ、全体の情報設計を崩す位置に無思考で足す。
  → 正しくは、意図（何を達成したいか）を汲み、既存の導線に馴染む形で実現する。

## 1. シンプルで“見慣れた”標準 iOS UI

- **原則**：装飾や独自操作を避け、一般的な標準 iOS の見た目・操作にする。plain / familiar。
- **なぜ**：ターゲット（20代）が迷わず使えること、"仕事感/管理感"を出さないこと（コアコンセプト§6）。
  奇抜さは学習コストと違和感を生む。
- **実装での守り方**：標準コンポーネント・標準ジェスチャ（スワイプ削除、シート、リスト）を第一選択に。
  フォントは system font（`.system(design: .default)`／`Theme.Font` はすべて SF にマップ済み）。
  達成時の演出は**トースト＋undo**のみ。**confetti / キラキラアニメは入れない**（README「Done trail」）。
- **破ってはいけない例**：紙＋クレヨン風の丸ゴシック/セリフ体を復活させる、達成時に紙吹雪を出す、
  独自のカスタムジェスチャを新設する。

## 2. 控えめで統一感のある表現

- **原則**：色は **1系統（green）** に統一し、その濃淡で構造（優先度の深さ）を表す。
  違いは**色数ではなく、形・長さ**で直感的に伝える。
- **なぜ**：多色は情報を散らかし"うるさい"。統一されたトーンが「そっと差し出す」プロダクト性格に合う。
- **実装での守り方**：構造的な意味づけは `Theme.green50–900` の濃淡で（例：優先度 高/中/低 = green700/500/300）。
  `sun*` / `peach*` は**限定的な装飾アクセントのみ**。優先度やカウントの差はチップの長さ・形・数で見せる。
- **破ってはいけない例**：優先度を赤/黄/青の3色で塗り分ける、タグごとに固有色を割り当てて虹色にする。

## 3. 本物の Liquid Glass（近似で代用しない）

- **原則**：フローティングのクローム（タブバー等）は **iOS 26+ の純正 `.glassEffect`** を使う。
  近似マテリアルで"それっぽく"代用しない。
- **なぜ**：本物の質感でこそ Apple 標準体験と地続きになる。近似は劣化コピーで、統一感原則にも反する。
- **実装での守り方**：`View.glass(in:)`（`Theme.swift`）を使う。iOS 26+ は本物の `.glassEffect(.regular, in:)`、
  それ未満は `.ultraThinMaterial`＋白ヘアラインへ**自動フォールバック**（＝旧OSの近似は許容だが、
  新OSで本物が使える箇所を近似で塗り替えるのは禁止）。対象は `CustomTabBar`・選択アクション円など。
- **破ってはいけない例**：iOS 26 でも `.ultraThinMaterial` を直書きしてガラス風を自作する。

## 4. 一括編集は Apple メール風の選択モード

- **原則**：複数選択・一括操作の UI は **Appleメール風**（編集 → 各行に選択○ → 下部ツールバーで一括操作）。
- **なぜ**：見慣れた標準パターン（原則1）で、誤操作が少なく学習不要。
- **実装での守り方**：`RowSelectCircle`（`Components.swift`、`CheckmarkShape`）＋選択状態＋下部バーの構成を踏襲。
- **破ってはいけない例**：長押しメニューや独自モーダルで一括編集を再発明する。

## 5. 追加フローの統一（モードを分けない）

- **原則**：「AI追加」「手入力追加」を**モードとして分けない**。URL の有無で自然に振る舞いが決まる、
  単一の追加フロー。AI は**ユーザーが未編集のフィールドだけ**を補完する。
- **なぜ**：モード選択は迷いを生む。ユーザーが主役で、AI は黙って下ごしらえするだけ（原則9）。
- **実装での守り方**：全経路（アプリ内 `AddEditSheet`／共有 `ShareComposeView`）が同一の `ItemForm` を使う。
  URL が入れば `Capture/` が候補を生成、未編集フィールドにのみ反映。触れたフィールドは上書きしない
  （"touched" 追跡）。
- **破ってはいけない例**：「AIで作る/自分で書く」の二択画面を最初に出す。ユーザーが直した値をAIが再上書きする。

## 6. 誤操作への配慮（先回りで守る）

- **原則**：破棄確認・undo・限界フィードバック・入力検証を、言われる前に入れる。
- **なぜ**：turnkey な品質。ユーザーが「やってしまった」と感じる瞬間を作らない。
- **実装での守り方**：破壊操作（削除・一括削除・タグ削除）には **undo トースト**（`pendingUndo`）。
  未保存の変更がある離脱には破棄確認。上限（カスタムタグ10・タグ3件）に達したら**その場でフィードバック**。
  入力は保存前に検証（空タイトル等）。
- **破ってはいけない例**：削除を確認・undo なしで即実行する。上限超過を黙って握りつぶす。

## 7. 意図を推測しない（自動生成の言い切り方）

- **原則**：自動生成テキストは**主題を体言止め**で出す。**動詞（＝ユーザーの意図）を足さない**。
- **なぜ**：「オーロラ」を見たいのか撮りたいのか調べたいのかは本人しか知らない。AIが意図を捏造すると外す。
- **実装での守り方**：LLM（`OnDeviceModel`、`@Guide` で体言止め指定）とルールベース（`RuleBasedCandidate`）の
  **両方で統一**。「〜する」「〜したい」を付けず、対象（名詞句）で止める。タイトルは30字上限・タグは最大3。
- **破ってはいけない例**：「オーロラを見に行く」と意図を勝手に補う。SNS投稿者名を主題に採用する。

## 8. AI は補助、ユーザーが主役／オンデバイス完結

- **原則**：AI の出力は常に**「候補」**。1タップで採用でき、削除せず**上書き**で自由に直せる。
  推論は**すべて端末内**で完結し、外部送信しない（バックエンドを持たない）。
- **なぜ**：コアコンセプト§7。依存しすぎない設計。プライバシーと"自分のリスト"感を守る。
- **実装での守り方**：URL候補は Apple Foundation Models（iOS 26 オンデバイス）→ 使えなければ
  ルールベースへ**フォールバック**。入力に使うのは端末内取得メタデータ（LinkPresentation 中心）のみ。
  タイトル欄は focus で全選択（`SelectAllTextField`）＝1打で上書き可。設定の自動分類 OFF を尊重。
- **破ってはいけない例**：分類やタイトル生成のためにサーバーAPIへテキストを送る。AIの結果を編集不可にする。
  ※注意：リンクのメタ取得自体は対象URLへの通信が発生するため、「一切通信しない/完全オフライン」とは
  **謳わない**（[release-runbook](../workflows/release-runbook.md) の注記）。「推論はオンデバイス」が正しい表現。

## 9. データを壊さない（寛容な永続化）

- **原則**：保存データは、部分的な破損や将来のスキーマ変更で**全消失させない**。
- **なぜ**：ユーザーの"いつか"の蓄積こそがこのアプリの資産。1件の不整合で全部失うのは最悪の裏切り。
- **実装での守り方**：`SharedStore`（`NSFileCoordinator`＋1世代バックアップ）、`LossyArray`/`LossyDocument`
  （1件破損はその1件だけスキップ）、`StoreLoad`（`.absent`/`.loaded`/`.unreadable` を区別し、読めない時は
  上書きせずバックアップ復旧）、`BucketItem` の寛容な `Codable`（新フィールドは既定値フォールバック）。
- **破ってはいけない例**：デコード失敗時にストア全体を空で上書きする。新フィールド追加で旧データが読めなくなる。

## この原則の使い方（AI向け）

1. コード変更の前に、関係する原則を確認する（`.claude/rules/*` がパスに応じて自動で該当分を出す）。
2. 実装後、`design-guardian` エージェント（`.claude/agents/`）で差分を原則に照らしてレビューする。
3. 原則同士が衝突したら、番号の小さい方（0 が最上位）を優先する。
4. どうしても原則を破る必要があると判断したら、**勝手に破らず** ADR 起票を提案し、依頼者の判断を仰ぐ。
